# Guide 05 — Generating packages

This is the **producer** side of the workflow: how to turn the source
tree into shippable artefacts (`.whl`, `.tar.gz`, `.exe`, `.zip`,
`.deb`, `.rpm`, `.pkg`). For the **consumer** side ("I have the
artefact, how do I install it?"), see [Guide 04](04-installing.md).

| Artefact | Built with | Section |
|---|---|---|
| Python wheel  | `python -m build --wheel` (drives scikit-build-core → CMake) | [§1 Wheel](#1-wheel) |
| Source dist (sdist) | `python -m build --sdist` | [§2 sdist](#2-sdist) |
| Windows ZIP / NSIS `.exe` | `cpack -G ZIP / NSIS` | [§3 Windows installers](#3-windows-installers-zip--nsis) |
| Linux `.deb` / `.rpm` / `.tar.gz` | `cpack -G DEB / RPM / TGZ` | [§4 Linux packages](#4-linux-packages-deb--rpm--tarball) |
| macOS `.pkg`  | `cpack -G productbuild` | [§5 macOS package](#5-macos-package-pkg) |

All artefacts land in `dist/` at the repo root.

## 1. Wheel

The wheel is built by **scikit-build-core**, a PEP 517 backend that
runs CMake under the hood and packages the install tree as a wheel.
Configuration lives in [`pyproject.toml`](../../pyproject.toml).

### 1.1 Install the build front-end (one time)

```powershell
python -m pip install --upgrade build
```

### 1.2 Build the wheel

```powershell
# Windows / Linux / macOS — same command everywhere:
python -m build --wheel --outdir dist
```

What happens internally:

1. `build` reads `pyproject.toml`, sees `build-backend = "scikit_build_core.build"`.
2. scikit-build-core creates a temporary `build/wheel/<tag>/` directory.
3. It runs `cmake` with `-DSKBUILD_PROJECT_NAME=pixelforge`, which
   makes [`CMakeLists.txt`](../../CMakeLists.txt) skip the SDK install
   block (we only want Python bits in the wheel).
4. It compiles the runtime DLL, the plugins, and `_pixelforge.pyd`.
5. It assembles the wheel layout described in
   [Guide 04 §1.4](04-installing.md#14-what-just-happened-on-disk).
6. It emits the `.whl` into `dist/`.

Final filename:

```
dist\pixelforge-0.1.0-cp313-cp313-win_amd64.whl
```

Decoded:

| Token       | Meaning                                  |
|-------------|------------------------------------------|
| `pixelforge`| project name from `pyproject.toml`       |
| `0.1.0`     | version                                  |
| `cp313`     | requires CPython 3.13                    |
| `cp313`     | the ABI the binary uses                  |
| `win_amd64` | platform: 64-bit Windows                 |

On Linux you'll get e.g. `…-cp313-cp313-linux_x86_64.whl` initially;
running `auditwheel repair` (only needed for PyPI) renames it to
`…-manylinux_2_28_x86_64.whl` and bundles in any external `.so`.

### 1.3 Build wheels for many Pythons at once (cibuildwheel)

```bash
pip install cibuildwheel
cibuildwheel --platform linux                # requires Docker
cibuildwheel --platform windows
cibuildwheel --platform macos
```

cibuildwheel reads the `[tool.cibuildwheel]` section of
`pyproject.toml` plus [`cibuildwheel.toml`](../../cibuildwheel.toml).
Output goes to `wheelhouse/`. This is what
[`.github/workflows/release.yml`](../../.github/workflows/release.yml)
runs in CI.

## 2. sdist

The source distribution is a tarball of the source tree, used by `pip`
when no wheel matches the user's platform.

```powershell
python -m build --sdist --outdir dist
```

Result:

```
dist\pixelforge-0.1.0.tar.gz   (~180 KB)
```

Contents are governed by `[tool.scikit-build].sdist.{include,exclude}`
in `pyproject.toml`. By default it includes everything needed to
re-run the full CMake build: source, headers, CMake scripts, third-party
sources, tests.

To verify what went in:

```bash
tar -tzf dist/pixelforge-0.1.0.tar.gz | head -30
```

## 3. Windows installers (ZIP + NSIS)

This is the CPack pipeline (Phase 7). It works on the install tree
produced by `cmake --install`, then wraps it as either a ZIP archive
or an NSIS executable installer.

### 3.1 Prerequisites

Install **NSIS** (only for the `.exe` installer; the ZIP needs nothing extra):

```powershell
winget install NSIS.NSIS
# Then make sure it's on PATH for the current shell:
$env:PATH = "C:\Program Files (x86)\NSIS;$env:PATH"
```

### 3.2 Build + package, end-to-end

```powershell
# 1. Configure + build (the same as Guide 01):
cmake --preset win-mingw
cmake --build --preset win-mingw

# 2. Run CTest as a gate (don't ship broken installers):
ctest --preset win-mingw --output-on-failure

# 3. Package — runs from inside the build directory:
Push-Location build\win-mingw
cpack -B ..\..\dist
Pop-Location
```

`cpack` reads the `CPACK_*` variables baked into `CPackConfig.cmake`
(generated by CMake at configure time) and produces every generator
listed in `CPACK_GENERATOR`, which on Windows is `ZIP;NSIS`.

Final artefacts:

```
dist\pixelforge-0.1.0-Windows-AMD64.zip   (~5.5 MB, portable)
dist\pixelforge-0.1.0-Windows-AMD64.exe   (~2.1 MB, NSIS installer)
```

### 3.3 Pick a single generator

```powershell
cpack -G ZIP  -B ..\..\dist     # only the ZIP
cpack -G NSIS -B ..\..\dist     # only the NSIS
```

### 3.4 Component selection

Both generators expose the three install components defined in
[`CMakeLists.txt`](../../CMakeLists.txt) (Runtime, SDK, Plugins). Users
pick which to install in the NSIS wizard or when extracting from the
ZIP. To build a per-component archive:

```powershell
cpack -G ZIP -B ..\..\dist -DCPACK_ARCHIVE_COMPONENT_INSTALL=ON
# Produces:
#   pixelforge-0.1.0-Windows-AMD64-Runtime.zip
#   pixelforge-0.1.0-Windows-AMD64-SDK.zip
#   pixelforge-0.1.0-Windows-AMD64-Plugins.zip
```

### 3.5 Test the installer locally

```powershell
$prefix = 'C:\Tools\pixelforge-test'
Remove-Item -Recurse -Force $prefix -ErrorAction SilentlyContinue
Start-Process -Wait -FilePath 'dist\pixelforge-0.1.0-Windows-AMD64.exe' `
              -ArgumentList "/S","/D=$prefix"
& "$prefix\bin\pixelforge.exe" --version
```

## 4. Linux packages (DEB + RPM + tarball)

Same recipe as §3, different generators. Run on a real Linux host (or
inside the Phase 8 Docker container, see [Guide 06](06-docker-builds.md)).

### 4.1 Prerequisites

```bash
sudo apt-get install -y rpm dpkg-dev   # both RPM and DEB tooling
```

### 4.2 Build + package

```bash
cmake --preset linux-gcc
cmake --build --preset linux-gcc
ctest --preset linux-gcc --output-on-failure

cd build/linux-gcc
cpack -B ../../dist                 # builds TGZ + DEB + RPM by default
```

Final artefacts:

```
dist/pixelforge-0.1.0-Linux-x86_64.tar.gz
dist/pixelforge_0.1.0_amd64.deb
dist/pixelforge-0.1.0-1.x86_64.rpm
```

### 4.3 Inspect a `.deb` before shipping

```bash
dpkg-deb -I dist/pixelforge_0.1.0_amd64.deb     # control file
dpkg-deb -c dist/pixelforge_0.1.0_amd64.deb     # file list with permissions
```

### 4.4 Inspect a `.rpm`

```bash
rpm -qpi  dist/pixelforge-0.1.0-1.x86_64.rpm    # info
rpm -qpl  dist/pixelforge-0.1.0-1.x86_64.rpm    # file list
rpm -qpR  dist/pixelforge-0.1.0-1.x86_64.rpm    # required deps
```

## 5. macOS package (`.pkg`)

```bash
brew install ninja cmake
cmake --preset linux-gcc        # macOS uses the linux-gcc preset (gcc=clang on macOS)
cmake --build --preset linux-gcc
ctest --preset linux-gcc --output-on-failure

cd build/linux-gcc
cpack -G "TGZ;productbuild" -B ../../dist
```

Final artefacts:

```
dist/pixelforge-0.1.0-Darwin.tar.gz
dist/pixelforge-0.1.0-Darwin.pkg
```

For App Store distribution you'd also `codesign` and `notarytool` —
see [`docs/future/improvements.md`](../future/improvements.md) §E.

## 6. One command does the lot, on every OS, in CI

The release workflow [`.github/workflows/release.yml`](../../.github/workflows/release.yml)
is the all-in-one. Push a tag:

```bash
git tag v0.1.0
git push --tags
```

…and GitHub Actions will produce, for that tag:

- 16 wheels (4 Pythons × 4 platforms)
- 1 sdist
- Windows `.exe` + `.zip`
- Linux `.deb` + `.rpm` + `.tar.gz`
- macOS `.pkg` + `.tar.gz`

…then attach all of them to a GitHub Release and (optionally) publish
the wheels to PyPI. See
[`docs/phases/phase-09-ci-cd.md`](../phases/phase-09-ci-cd.md) for the
full breakdown.

## 7. Where everything lands — final cheat sheet

After a complete local build + package on Windows MinGW:

```
pixelforge/
├── build/win-mingw/                  ← intermediate, ~140 MB
│   ├── bin/                          ← built CLI + DLL
│   ├── lib/                          ← built .a + import lib
│   ├── plugins/                      ← built .dll plugins
│   ├── python_modules/               ← built .pyd
│   ├── _CPack_Packages/              ← cpack staging area
│   └── …
└── dist/                             ← shippable, ~10 MB total
    ├── pixelforge-0.1.0.tar.gz                       (sdist)
    ├── pixelforge-0.1.0-cp313-cp313-win_amd64.whl    (wheel, 1.2 MB)
    ├── pixelforge-0.1.0-Windows-AMD64.zip            (5.5 MB)
    └── pixelforge-0.1.0-Windows-AMD64.exe            (2.1 MB)
```
